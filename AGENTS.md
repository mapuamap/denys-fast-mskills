### Architecture decision (important)
Skills + agents are **plugin-level / global** — they become available in every project once the plugin is installed. `/m_setup` installs only the **project-local enforcement scaffolding** (rules + hooks + `ai-check.sh` + settings), because hooks/rules must live per-repo. Do not move the `m_code_*` skills back into the payload.

### Two planning paths coexist by design
- Lightweight: `infra.md` / `deploy.md` context files driven by `/m_infr*` and `/m_deploy*`.
- Formal: the `m_plan` artifact pipeline (`.m_plan/<slug>/01..09`, where `09_verification.md` is the completion oracle).
Keep both; the README explains when to use which.

### m_verify removed (v3.0.0)
The whole `m_verify` feature-verification surface was removed at the owner's request: the `m_verify` skill, the `m_verify-repair` agent, the `.m_verify/` ledger, and the Stop hook that silently fed it. Don't reintroduce a hook (or any other mechanism) that makes the agent record "done" items into a `.m_verify` ledger — that's the exact behavior that was cut. Verification now lives only in `m_plan`'s `09_verification.md` oracle and the per-repo `m_code_framework` checks.

### m_plan completion — `09_verification.md` is the oracle (autodrive hook removed in v2.12.0)
`m_plan` / `m_plan_implement` decide "done" from `09_verification.md` (zero open `[ ]` rows), not the agent's judgement. The skills walk the plan step by step and update `09` as they go; for a hands-off long run, the user opts into `/goal` mode (the skills emit the `/goal` line on request).
- **Removed:** the `hooks/plan_completion_driver.py` Stop hook (and its `.m_plan/<slug>/.run.json` arming in `m_plan_implement` Phase B/C and `m_plan` Phase 2/3) was deleted in v2.12.0 — it drove runs to completion automatically but behaved incorrectly. The skills no longer write or read `.run.json`. If you reintroduce any auto-completion mechanism, do **not** resurrect a global Stop hook that blocks stopping; prefer the explicit, user-invoked `/goal` path.

### End-of-turn hook — «Что дальше»/«Задача» summary only (restored v3.1.0)
History of this hook: branded **widget** removed v2.11.0; autodrive driver removed v2.12.0; the «Что дальше»/«Задача» **text summary** lived in v2.13.0, removed v2.14.0; the whole `turn_summary_widget.py` (stats line **and** `.m_verify` ledger feed) deleted v3.0.0; then in **v3.1.0** the owner asked for the text summary back, so a lean `hooks/turn_summary.py` was added that emits **only** the two blocks.
- What it does now: on a Stop event, parse the transcript for the last human turn; if `tools_total >= MIN_TOOLS` (env `CLAUDE_SUMMARY_MIN_TOOLS`, default 2) emit `decision:block` whose reason instructs the agent to write «Что дальше» (what done + next step) and «Задача» (one-sentence restate of the request). Trivial turns and the second firing (`stop_hook_active`) produce no output. Fail-open.
- **Deliberately NOT here:** no `show_widget`/widget, no one-line stats `systemMessage`, no `.m_verify` ledger/`build_ledger_reason`. Do not re-add any of those — the owner removed the stats line and the whole `m_verify` verification surface in v3.0.0 and wants only the summary.
- **Hook-logic changes** (anything in `turn_summary.py`) need a normal plugin release: bump `plugin.json` + commit/push + the owner reinstalls/updates the plugin.
- **Legacy assets** `docs/widget/m_widget.js` and `docs/widget/index.html` are unreferenced dead history — safe to delete later.

## Conventions

- **Language:** everything user-facing is **English** (READMEs, SKILL/agent bodies, landing page). (Originals were partly Russian; that was translated on first release — keep new content English.) **Deliberate exception:** the Stop-hook *summary instruction* in `hooks/turn_summary.py` (`build_reason()`) is Russian — that's the owner's personal hook text («Что дальше»/«Задача»); keep new hook-instruction text consistent with that file.
- **Commands** carry frontmatter `disable-model-invocation: true` (user-triggered only). Keep it on new commands unless you explicitly want model auto-invocation.
- **Skill** = a directory under `skills/<name>/` with `SKILL.md` (YAML frontmatter `name` + `description`, then body). Bundled resources go in subdirs (`templates/`, `evals/`, `BLOCKERS.md`, …).
- **Naming.** A skill's `/command` is its **directory name** — keep the `name:` frontmatter identical to the directory (underscores, e.g. `m_code_refactor`). **Agents are different:** their `name:` must use **hyphens** (`m_code-context-scout`, `m_code-test-runner`, `m_code-architecture-reviewer` — the files use underscores but `name:` is hyphenated), and that hyphenated name is the `subagent_type` skills reference when they delegate. Don't "fix" agent names to underscores — it breaks delegation.
- **Skill→agent wiring.** The `m_code_*` skills delegate heavy sub-steps to the three agents: `m_code-context-scout` (wide read-only scans), `m_code-test-runner` (noisy build/test/lint runs), `m_code-architecture-reviewer` (independent boundary review). When editing these skills, keep those references — they exist so verbose work stays out of the main context. `m_plan` / `m_plan_implement` reference the scout / test-runner too.
- **Invocation modes.** `m_code_*` skills are user-invoked (`disable-model-invocation: true`) — deliberate precision tools. That flag drops the description from the model's context, so write their descriptions as crisp human menu items, not auto-trigger bait. `m_plan` / `m_plan_implement` are model-invocable on purpose (they're the front door).
- **No secrets, no machine-specific paths.** Never commit `H:\...`, `~/bin`, IPs, hostnames, tokens, SSH key names. The one allowed literal is `id_rsa` inside `m_code_framework/hooks/m_code_protect_files.py` — that's a *protection blocklist*, not a secret.
- `m_setup` must stay portable: it shells out to `${CLAUDE_PLUGIN_ROOT}/m_code_framework/install.sh`, which self-locates via `BASH_SOURCE`. Don't hardcode an install source.

## Editing workflow

1. Edit the relevant file under `skills/`, `commands/`, `agents/`, or `m_code_framework/`.
2. If you add/rename a skill or command, update `README.md` and (if structure changed) this file.
3. If you changed any skill, rebuild the Desktop/web pack: `python scripts/build_skills_pack.py` (regenerates `docs/skills-pack/*.zip`).
4. Bump `version` in `.claude-plugin/plugin.json` for user-visible changes (semver).
5. Validate before committing (see below).
6. Commit + push to `main`. Tag releases if you bump a minor/major version.
