### Architecture decision (important)
Skills + agents are **plugin-level / global** — they become available in every project once the plugin is installed. `/m_setup` installs only the **project-local enforcement scaffolding** (rules + hooks + `ai-check.sh` + settings), because hooks/rules must live per-repo. Do not move the `m_code_*` skills back into the payload.

### Two planning paths coexist by design
- Lightweight: `infra.md` / `deploy.md` context files driven by `/m_infr*` and `/m_deploy*`.
- Formal: the `m_plan` artifact pipeline (`.m_plan/<slug>/01..09`, where `09_verification.md` is the completion oracle).
Keep both; the README explains when to use which.

### m_verify — hook-fed feature-verification manager
A third, always-on verification surface that is **separate** from `m_plan`'s `09_verification.md` by design.
- **Ledger:** `.m_verify/pending.md` in the target project (confirmed items archived to `.m_verify/confirmed.md`). Canonical format = flat list of `## ` sections keyed by a `key:` field; field names (`status`/`who`/`how`/…) are English. **This format is a contract shared between two producers/consumers — keep them in sync:** the Stop hook (`hooks/turn_summary_widget.py`, `build_ledger_reason()`) *appends* candidates, and `skills/m_verify/SKILL.md` *curates* them. Change one → change the other.
- **Feed gate:** the hook only appends on turns where code changed (`CODE_TOOLS` = Edit/Write/MultiEdit/NotebookEdit present in `tools_by_name`). The append is a *silent* instruction (file IO only, nothing in chat) tacked onto the end of the `build_reason()` block as step 3, after the «Что дальше»/«Задача» text blocks. (v2.11.0 removed the branded widget; v2.13.0 restored the text blocks that had been dropped with it — the ledger step rode along through both.)
- **Delegation:** `m_verify` (interactive, main context) delegates AI checks to `m_code-test-runner` and fixes to **`m_verify-repair`** (run in background so the user keeps flowing). Repair edits the working tree but **never commits** — that's the user's call.

### m_plan completion — `09_verification.md` is the oracle (autodrive hook removed in v2.12.0)
`m_plan` / `m_plan_implement` decide "done" from `09_verification.md` (zero open `[ ]` rows), not the agent's judgement. The skills walk the plan step by step and update `09` as they go; for a hands-off long run, the user opts into `/goal` mode (the skills emit the `/goal` line on request).
- **Removed:** the `hooks/plan_completion_driver.py` Stop hook (and its `.m_plan/<slug>/.run.json` arming in `m_plan_implement` Phase B/C and `m_plan` Phase 2/3) was deleted in v2.12.0 — it drove runs to completion automatically but behaved incorrectly. The skills no longer write or read `.run.json`. If you reintroduce any auto-completion mechanism, do **not** resurrect a global Stop hook that blocks stopping; prefer the explicit, user-invoked `/goal` path.

### End-of-turn summary — text blocks kept, widget removed (v2.11.0 / v2.13.0)
The branded in-chat **widget** (logo card + the `show_widget` render step) was **removed at the owner's request** in v2.11.0. In that same change the plain-text «Что дальше»/«Задача» notes were dropped too — but the owner only wanted the *widget* gone, so they were **restored in v2.13.0**. What `turn_summary_widget.py` does now:
- Every non-empty turn → a lightweight one-line ANSI `systemMessage` (project, token/cost/time stats) via `one_line_summary()`.
- Real-work turns (`tools_total >= MIN_TOOLS_FOR_WIDGET`, default 2) → `decision:block` whose reason (`build_reason()`) is two plain-text blocks: «Что дальше» (what was done + next step) and «Задача» (one-sentence restatement of the user's request, rephrased from the last human prompt). **No widget, no `show_widget`.**
- Code-change turns → the silent `.m_verify` ledger step (`ledger_block()`, step 3) is appended to that same reason. Trivial turns (< MIN tools) → one-liner only, no block.
- **Hook-logic changes** (anything in `turn_summary_widget.py`) need a normal plugin release: bump `plugin.json` + commit/push + the owner reinstalls/updates the plugin.
- **Legacy assets** `docs/widget/m_widget.js` and `docs/widget/index.html` are **unreferenced** by the hook (no `WIDGET_URL`, no jsDelivr fetch) — left for history, safe to delete later. The env knob is still named `CLAUDE_WIDGET_MIN_TOOLS` / `MIN_TOOLS_FOR_WIDGET` for back-compat though there's no widget.

## Conventions

- **Language:** everything user-facing is **English** (READMEs, SKILL/agent bodies, landing page). (Originals were partly Russian; that was translated on first release — keep new content English.) **Deliberate exception:** the Stop-hook *instructions* in `hooks/turn_summary_widget.py` (`build_reason()` text blocks «Что дальше»/«Задача» and the `ledger_block()` step) are Russian — that's the owner's personal hook text; keep new hook-instruction text consistent with that file. The `.m_verify` ledger *field names* stay English (parsing contract); entry titles/notes follow the working language.
- **Commands** carry frontmatter `disable-model-invocation: true` (user-triggered only). Keep it on new commands unless you explicitly want model auto-invocation.
- **Skill** = a directory under `skills/<name>/` with `SKILL.md` (YAML frontmatter `name` + `description`, then body). Bundled resources go in subdirs (`templates/`, `evals/`, `BLOCKERS.md`, …).
- **Naming.** A skill's `/command` is its **directory name** — keep the `name:` frontmatter identical to the directory (underscores, e.g. `m_code_refactor`). **Agents are different:** their `name:` must use **hyphens** (`m_code-context-scout`, `m_code-test-runner`, `m_code-architecture-reviewer`, `m_verify-repair` — note the file is `agents/m_verify_repair.md` but `name:` is `m_verify-repair`), and that hyphenated name is the `subagent_type` skills reference when they delegate. Don't "fix" agent names to underscores — it breaks delegation.
- **Skill→agent wiring.** The `m_code_*` skills delegate heavy sub-steps to the three agents: `m_code-context-scout` (wide read-only scans), `m_code-test-runner` (noisy build/test/lint runs), `m_code-architecture-reviewer` (independent boundary review). When editing these skills, keep those references — they exist so verbose work stays out of the main context. `m_plan` / `m_plan_implement` reference the scout / test-runner too.
- **Invocation modes.** `m_code_*` skills are user-invoked (`disable-model-invocation: true`) — deliberate precision tools. That flag drops the description from the model's context, so write their descriptions as crisp human menu items, not auto-trigger bait. `m_plan` / `m_plan_implement` are model-invocable on purpose (they're the front door).
- **No secrets, no machine-specific paths.** Never commit `H:\...`, `~/bin`, IPs, hostnames, tokens, SSH key names. The one allowed literal is `id_rsa` inside `m_code_framework/hooks/m_code_protect_files.py` — that's a *protection blocklist*, not a secret.
- `m_setup` must stay portable: it shells out to `${CLAUDE_PLUGIN_ROOT}/m_code_framework/install.sh`, which self-locates via `BASH_SOURCE`. Don't hardcode an install source.

## Editing workflow

1. Edit the relevant file under `skills/`, `commands/`, `agents/`, or `m_code_framework/`.
2. If you add/rename a skill or command, update `README.md` and (if structure changed) this file.
3. If you changed any skill, rebuild the Desktop/web pack: `python scripts/build_skills_pack.py` (regenerates `docs/skills-pack/*.zip`).
4. Bump `version` in `.claude-plugin/plugin.json` for user-visible changes (semver).
5. Validate before committing (see below).
6. Commit + push to `main`. Tag releases if you bump a minor/major version.
